# 网文生成大模型 — 项目执行规划

## Context

**目标**: 构建基于大模型的都市文娱小说生成系统，作为 AI 岗位实习申请作品集。通过 SFT 微调使基座模型掌握都市文娱的文风与叙事模式，结合双路 RAG（文风参考 + 知识检索）增强生成质量，通过约束系统强制执行网文结构规范（黄金三章、单元化叙事、高潮节奏）。全程使用高质量外部 LLM（DeepSeek V4 Pro / GPT-5.5 等）辅助数据标注、内容分类和生成校验。

**核心决策**:

| 决策项 | 选择 |
|--------|------|
| 基座模型 | **Qwen3-30B-A3B**（30B 总参 / 3B 激活，中文能力强，4×A5000 可宽松微调） |
| 备选模型 | DeepSeek V4 Flash（284B MoE，硬件条件允许时切换） |
| 题材 | **都市-文娱**（娱乐圈/导演/歌手/文抄公等重生或系统流） |
| 微调方式 | Qwen3-30B-A3B: LoRA/全参数微调（4×A5000 96GB 充裕） |
| RAG | 双路：文风参考 + 结构化知识（人物/作品/行业梗） |
| 数据标注 | 外部高质量 LLM（DeepSeek V4 Pro）辅助场景分类、高潮标注、质量筛选 |
| 约束系统 | 大纲规划器 + 章节生成器 + LLM 校验器（由外部强大模型执行校验） |
| 硬件 | 4× NVIDIA A5000（24GB×4 = 96GB 总显存） |
| 周期 | 2 个月出 Demo |

## 硬件可行性评估

### 推荐方案：Qwen3-30B-A3B（主方案）

Qwen3-30B-A3B 是 30B 总参 / 3B 激活的 MoE 模型，中文能力出色，在 4×A5000（96GB）上可宽松微调：

| 精度/场景 | 推理 | LoRA 微调 | 全参数微调 |
|-----------|------|-----------|-----------|
| FP16（~60GB 磁盘） | ~60GB | ~70GB | ~90GB+ZeRO-3 |
| INT4 量化（~17GB 磁盘） | ~17GB | ~30GB | N/A |

**结论：4×A5000 对 Qwen3-30B-A3B 完全够用。** LoRA 微调轻松运行，甚至可尝试全参数微调（DeepSpeed ZeRO-3 + CPU Offload）。这是本项目的**默认方案**。

> **⚠️ 激活参数量的隐忧**: Qwen3-30B-A3B 虽然总参数量 30B，但每次推理仅激活 3B 参数（MoE 架构）。3B 激活参数对网文生成是否足够，是本方案最大的未知数。网文需要的不只是知识广度（MoE 的优势），更重要的是**长文本连贯性、节奏控制、文风一致性**，这些更依赖单次前向的推理深度（激活参数量）。Qwen3-14B MVP 验证阶段的一个核心目标就是对比 14B Dense 与 30B MoE 的生成质量——如果 14B Dense 的连贯性明显更好，说明 3B 激活确实不够，需要重新评估模型选择。

### 备选方案：DeepSeek V4 Flash

对于生成质量要求更高的场景，可将 DeepSeek V4 Flash（284B MoE / 13B 激活 / 1M 上下文）作为备选基座。但其硬件门槛显著更高：

| 精度/场景 | 推理 | QLoRA 微调 |
|-----------|------|-----------|
| INT2 量化（~75GB 磁盘） | ~96GB | **~110GB（超出 96GB）** |

即使 INT2 量化 + CPU Offload + ZeRO-3 极限组合，训练吞吐极低（可能 <1 token/s），且 MoE + INT2 + QLoRA 的收敛性缺乏公开验证。该方案需要硬件升级（≥2×H100 或租用云端 GPU）才具实操性，当前阶段仅作为远期扩展方向保留。

### 风险控制：Qwen3-14B 先行验证

在大规模微调 Qwen3-30B-A3B 之前，先用 Qwen3-14B（Dense 模型，单张 A5000 即可运行）在 1 周内跑通全流程 MVP。目的不是比较模型质量，而是**验证数据管线、约束系统和 RAG 的技术可行性**——用最小成本发现问题、迭代方案，降低正式训练阶段的返工风险。

> **硬件策略**: Qwen3-14B 先行验证管线（1周）→ Qwen3-30B-A3B 正式微调（主方案）→ DeepSeek V4 Flash（远期扩展，需硬件升级）

## 都市文娱题材特征分析

在进入技术方案前，先明确都市文娱小说的核心特征，这直接影响数据构造、Prompt 设计和约束规则。

### 文风特点
- **轻松诙谐、爽感驱动**: 语言口语化、有梗、节奏快，拒绝沉重说教
- **半架空与现实交织**: 要么是华娱（真实明星/事件），要么是平行世界（文抄公流）
- **专业感+接地气**: 行业细节要有真实感（如拍电影流程、综艺录制），但人物行为要有人情味

### 叙事结构
- **单元化叙事**: 以"项目"为单位（拍一部电影、上一档综艺、发一张专辑），每个单元有完整的铺垫→高潮→收尾
- **三线并进**: 事业线（文娱成就）+ 感情线（恋爱/修罗场）+ 日常线（轻松互动/配角情景剧）
- **金手指模式**: 重生先知、系统任务、文抄公是主流，金手指需"模糊化"处理（不详细解释机制）

### 黄金三章（都市文娱版）
1. **第一章**: 穿越/重生设定 + 当前困境（如：平行世界的落魄歌手/导演）+ 金手指激活
2. **第二章**: 金手指初试（如：搬运第一首歌/第一部电影）+ 小范围引起关注
3. **第三章**: 第一次冲突/打脸 + 更大机遇降临 + 悬念钩子

### 高潮节奏（都市文娱版）
- **每 5 章小高潮**: 一个"单元"完结（一部作品发布/一次综艺播出/一次打脸成功）
- **每 10 章大高潮**: 重大事业突破（获奖/票房大爆/行业地位跃升）+ 感情线重大进展

## Phase 1: 数据管线（第 1-2 周）

### 任务 1.1 — 都市文娱小说数据采集

- **目标源**:
  - 起点中文网、番茄小说等平台的都市-娱乐分类热门作品
  - 重点作者: 青衫取醉、错哪儿了、尝谕、幼儿园一把手等都市文娱代表作家
  - 公开的 txt 下载资源站、论坛分享帖
  - ⚠️ **仅限个人学习研究，严禁传播或商用**
- **工具**: Python `httpx` + `selectolax` 异步爬虫，或使用已有 txt 合集
- **规模目标**: 30-50 本都市文娱题材小说，覆盖 3-5 位不同作者
- **产出**: `data/raw/` 下按 `{作者}/{书名}.txt` 组织

### 任务 1.2 — 数据清洗与结构化

- **清洗**: 去广告水印、统一 UTF-8、章节切分（正则匹配"第X章"）
- **都市文娱专项处理（使用 DeepSeek V4 Pro / GPT-5.5 等高质量 LLM）**:
  - 识别并标记场景类型: `日常互动` / `作品发布` / `综艺录制` / `商战谈判` / `感情戏` / `打脸场面`
  - 标注叙事节奏: `铺垫章` / `小高潮章` / `大高潮章` / `过渡章` / `收尾章`
  - 抽取结构化实体: 主角名、金手指类型（重生/系统/文抄公）、关键作品名、CP 名
  - 标注三线覆盖: 每章自动标注事业线/感情线/日常线的覆盖情况
  - 质量评分: 对每章做 1-5 分的文风质量评分，低于 3 分的样本降权或剔除
- **工具**: Python `httpx` + 高质量 LLM API + `jionlp`（辅助实体抽取）
- **Prompt 设计**: 针对每个标注任务设计专门的 few-shot prompt，利用强大 LLM 的理解能力做精细化标注，而非关键词匹配
- **标注策略——按任务主观性分层**:
  - **客观/半客观任务**（LLM 独立标注）: 场景类型分类、实体抽取、字数统计。这类任务 LLM 一致率预计 ≥85%（边界模糊时 LLM 也会有分歧），可全量自动标注。
  - **主观任务**（LLM 初筛 + 人工复核）: 高潮判断、文风质量评分。这类任务边界模糊，LLM 倾向于集中打分（3-4 分），区分度不足。采用"LLM 初筛 → 人工对边界 case 纠偏"的混合策略。
- **人工抽检**: 对 LLM 标注结果抽检 300 条（覆盖全部标注维度，主观任务 oversample），计算标注一致率（客观任务目标 ≥85%，主观任务目标 ≥75%）
- **产出**: `data/clean/` 下按 `{书名}/chapter_{N}.txt` 组织，附带 `metadata.json`（含 LLM 标注的场景类型、节奏标注、实体、质量分）

### 任务 1.3 — SFT 训练数据构造

- **格式**: 使用 Qwen3 原生 tokenizer 的 `apply_chat_template()` 自动生成训练格式，**不要硬编码 ChatML**。硬编码的 token 序列可能与 Qwen3 的 tokenizer 不一致，导致训练-推理行为异常。
  ```python
  # 正确做法：用 tokenizer 自动格式化
  messages = [
      {"role": "system", "content": "你是一个都市文娱小说作家，擅长轻松诙谐的文风..."},
      {"role": "user", "content": "{指令}"},
      {"role": "assistant", "content": "{生成的章节/段落}"}
  ]
  text = tokenizer.apply_chat_template(messages, tokenize=False)
  ```
  Qwen3 系列内部使用 ChatML 风格 tokens，但具体 token ID 和特殊 token 序列需通过官方 tokenizer 生成，不应手写。
- **任务类型构成**:

  | 任务类型 | 占比 | 示例 |
  |---------|------|------|
  | 章节续写 | 30% | 给前 3 章 → 生成第 4 章 |
  | 单元创作 | 25% | 给单元大纲（如"主角要拍一部小成本悬疑片"）→ 生成完整单元 |
  | 场景生成 | 20% | 给场景类型+人物 → 生成对应场景（打脸/综艺/日常） |
  | 风格模仿 | 15% | 给指定作者的 3 个段落 → 模仿其风格写新场景 |
  | 大纲扩展 | 10% | 给一句话梗概 → 生成 10 章大纲（含高潮标注） |

- **五种任务类型的数据构造逻辑**（从清洗后的章节自动生成训练样本）:

  | 任务类型 | 构造方法 | 来源 |
  |---------|---------|------|
  | 章节续写 (30%) | 同一本书取连续 N 章，前 N-1 章拼接为 user prompt（含前情），第 N 章为 assistant target | 同书连续章节 |
  
  > **章节续写上下文控制**: 3 章×3000 字 ≈ 15,000 tokens（中文），可能超出 8192 训练上限。处理策略: (1) N 优先取 2（即给前 1 章续写下一章），上下文更宽裕；(2) 如需 N≥3，将前 N-1 章每章截断至最后 1000 字（保留最近的上下文），保证 user prompt ≤5000 tokens；(3) 优先选择字数较短的章节（2000-2500 字）作为训练样本。
  | 单元创作 (25%) | 利用 Phase 1.2 的"节奏标注"识别单元边界（铺垫→高潮→收尾），将单元大纲（LLM 自动总结）作为 user prompt，单元全文作为 target | 同书同一单元 |
  | 场景生成 (20%) | 取 LLM 标注的场景类型标签 + 该场景前 200 字（设定人物和情境）作为 user prompt，该场景全文作为 target | 同书同一场景 |
  | 风格模仿 (15%) | 同作者随机取 3 段短文（各 200-400 字）作为 few-shot 注入 user prompt，另一段作为 target | 同作者跨书 |
  | 大纲扩展 (10%) | 用 LLM 将已有章节反向总结为大纲，取前 60% 大纲作为 user prompt，后 40% 大纲+内容作为 target | 跨书通用 |

  五种类型的 user prompt 需要设计不同的 system prompt 前缀和指令模板，避免模型混淆任务。

- **数据量目标**: 6,000-10,000 条（质量优先，不必追求数量上限）
- **弹性底线**: 如果 LLM 质量评分后有效样本不足 6,000 条，不要凑数。低质量样本对微调是负作用。可接受 4,000-5,000 条高质量样本，配合更低的 learning rate（1e-4）和更多 epoch（3-4）来补偿数量不足。
- **质量筛选**:
  - 规则过滤: 去除字数<500 或 >5000 的异常样本
  - LLM 质量评分: 使用 DeepSeek V4 Pro 对每条样本做 1-5 分质量评估（文风匹配度、结构完整性、趣味性），剔除 <3 分的样本
  - 人工抽检 300 条（覆盖全部 5 种任务类型，每种 ≥50 条）
- **产出**: `data/sft/train.jsonl`, `data/sft/val.jsonl` (9:1 分割)

### 任务 1.4 — RAG 知识库构建

**文风检索库**:
- Chunk 粒度: 200-500 字段落
- Embedding: `BAAI/bge-large-zh-v1.5`（中文 SOTA）
- 向量库: `ChromaDB`
- 元数据标签（均由 DeepSeek V4 Pro 自动标注）: 作者、场景类型（打脸/综艺/日常/感情/商战/发布）、节奏标注（铺垫/小高潮/大高潮/过渡/收尾）、情绪（爽/甜/虐/搞笑）、质量分（1-5）
- 特殊索引: "装逼打脸"名场面独立标注（LLM 识别），高权重检索

**知识检索库**（都市文娱专属）:
- **人物知识图谱**: 主角人设（职业/性格/金手指）、CP 关系、配角群像
- **娱乐圈知识库**: 常见综艺类型、电影制作流程、音乐产业术语、榜单体系
- **作品库**: 文抄公常用"搬运"作品列表（电影/歌曲/综艺模式）
- **梗库**: 都市文娱高频梗、流行语、经典桥段模板
- 存储: `ChromaDB` + `sqlite` 混合（结构化知识用 sqlite，语义检索用 ChromaDB）

### 任务 1.5 — 数据管线小规模验证（Week 1 末尾，在投入全量 API 费用前）

在全量 LLM 标注和 SFT 数据构造（预估 API 费用 $100-200）之前，先用最小成本验证数据格式和训练管线的正确性：

- 取 **2 本书 × 各 50 章 = 100 章**作为验证样本
- 用 DeepSeek V4 Pro 标注 100 章（费用 ~$3-5）
- 从标注结果中手工构造 **200 条 SFT 样本**，覆盖全部 5 种任务类型
- 用 Qwen3-14B 做一次快速 LoRA 微调（~30 分钟），验证:
  - 数据格式是否正确（tokenizer 输出是否符合预期）
  - 五种任务类型的比例是否合理（模型是否会偏向某种输出模式）
  - 训练 pipeline 是否能正常运行
- **通过标准**: 模型能完成一次简单的续写任务，不报错，不输出乱码

> 此验证与 Phase 2 开头的 Qwen3-14B MVP 互补：1.5 验证**数据格式正确性**（小规模），2.0 验证**全链路可行性**（功能级）。

## Phase 2: SFT 微调（第 2-4 周）

### Week 2 前半：Qwen3-14B 先行验证

在正式微调 Qwen3-30B-A3B 之前，用 Qwen3-14B（Dense 模型，单张 A5000 即可运行）快速跑通全流程：
- 用 500 条 SFT 样本做 LoRA 微调（半天内完成）
- 验证"数据→训练→RAG→约束→生成"全链路通畅
- 用最小成本发现管线问题、迭代方案，降低正式训练阶段的返工风险
- **关键对比实验**: 用相同 prompt（同一章的续写任务，约 2000 tokens 上下文）对比 Qwen3-14B Dense（14B 激活）和 Qwen3-30B-A3B（3B 激活）的生成质量。评估维度: 实体一致性（主角名是否贯穿）、情节连贯性（是否记得前文的设定）、文风统一性。用 DeepSeek V4 Pro 做盲评打分（1-5 分，3 个维度各一分数），取均值比较。
  
  **决策逻辑**（实验后按结果分流）:
  - **30B MoE 明显优于 14B Dense**（综合分差距 >0.5）→ 继续 Qwen3-30B-A3B 主方案，3B 激活担忧解除
  - **两者接近**（差距 ≤0.5）→ 仍用 Qwen3-30B-A3B（MoE 推理成本更低，总参数大带来的知识广度优势在微调后可能释放）
  - **14B Dense 明显优于 30B MoE**（差距 >0.5）→ 3B 激活确认不足。改用 Qwen3-14B Dense 作为主方案，同时将 Qwen3-30B-A3B 的全参数微调方案调整为 Qwen3-14B 全参数微调（4×A5000 更充裕，可开更大 batch size 和更长上下文）

## Phase 3: RAG 检索系统（第 4-5 周，可与 Phase 2 并行）

### 任务 3.1 — 文风检索（Style RAG）

- **检索策略**: 用户输入场景描述 → 编码 → 检索相似都市文娱段落
- **都市文娱专属优化**: 基于 LLM 预标注的场景标签和节奏标注进行过滤式检索
  - "打脸"场景 → 限定检索 LLM 标注的"装逼打脸"名场面索引
  - "综艺"场景 → 限定检索场景类型=`综艺录制` 的 chunk
  - "日常"场景 → 限定检索场景类型=`日常互动` 的 chunk
  - 高潮节点 → 优先检索节奏标注=`大高潮章`/`小高潮章` 的段落
- **使用方式**: 检索 Top-3 段落作为 few-shot 示例注入 system prompt
- **评估**: 检索结果与目标场景匹配率 ≥70%

### 任务 3.2 — 知识检索（Knowledge RAG）

- **持续知识维护**: 每生成一章后更新
  - 新增人物 → 更新人物表
  - 新增作品 → 更新作品库
  - 更新前情摘要（用 LLM 自动总结上一章）
- **检索时机**: 生成新章节前，根据当前大纲节点检索相关人物、设定、前情
- **冷启动策略（第 1 章生成时知识库为空）**:
  - 第 1 章生成前，先调用大纲规划器（Phase 4.1）生成完整大纲 JSON
  - 从大纲 JSON 的 `golden_three` 和 `units[0]` 字段提取初始知识:
    - `{summary}` → 填入用户梗概（前情为空，用梗概替代）
    - `{characters}` → 从大纲中提取初始角色（主角名、金手指、初始身份；配角在第 1 章可能尚未出场但可预填）
    - `{world_settings}` → 从大纲和用户输入中提取时代背景（如"2008年平行世界娱乐圈"）
  - 第 2 章起由 `knowledge_updater` 根据已生成内容自动增量维护所有字段
- **存储**:
  ```
  data/knowledge/
  ├── characters.json     # {"name": "林风", "role": "主角", "career": "导演",
  │                         "golden_finger": "重生2008", "traits": ["幽默", "腹黑"]}
  ├── works.json          # {"title": "失恋33天", "type": "电影", "result": "票房黑马"}
  ├── world_settings.json # {"era": "2008-2015", "industry": "娱乐圈", "parallel": true}
  ├── industry_kb.json    # 电影/音乐/综艺行业知识
  └── summary.txt          # 每章自动生成的前情提要
  ```

### 任务 4.1 — 大纲规划器

**输入**: 用户梗概（200-500 字的故事设定）

**输出**: 结构化大纲 JSON

```json
{
  "total_chapters": 50,
  "golden_three": [
    {"ch": 1, "hook": "重生2008+发现金手指+原身困境"},
    {"ch": 2, "hook": "金手指初试+小范围关注+埋下第一个冲突"},
    {"ch": 3, "hook": "第一次打脸成功+更大机遇+悬念钩子"}
  ],
  "units": [
    {
      "id": 1,
      "chapters": "4-8",
      "project": "拍摄第一部电影《失恋33天》",
      "mini_climax_ch": 5,
      "major_climax_ch": null,
      "lines": {
        "career": "低成本拍摄→遭遇撤资危机→口碑逆袭",
        "romance": "与女主角的初见与暧昧",
        "daily": "剧组日常搞笑+配角互动"
      }
    },
    {
      "id": 2,
      "chapters": "9-13",
      "project": "电影上映+票房奇迹",
      "mini_climax_ch": 10,
      "major_climax_ch": 13,
      "lines": {
        "career": "排片困境→口碑发酵→票房逆袭→行业震惊",
        "romance": "关系升温+修罗场",
        "daily": "爆红后的烦恼+朋友反应"
      }
    }
  ]
}
```

**关键约束**:
- 黄金三章: ch1-3 必须覆盖重生+金手指+第一次打脸
- 每 5 章: 一个小高潮（单元完结或重大进展）
- 每 10 章: 大高潮 + 感情线里程碑

### 任务 4.2 — 章节生成器

- **工作流**:
  1. 根据大纲节点确定当前章节的目标
  2. 调用 RAG 检索获取文风参考和知识上下文
  3. 构造融合 Prompt → 调用微调模型生成
  4. 后处理校验

- **字数控制**: Prompt 要求 + 生成后统计，不足自动续写，超出截断至最后一个完整段落
- **结尾钩子**: 强制检查章节最后 3 句是否包含悬念元素（疑问句/转折/预告），不满足则要求重写最后一段

### 任务 4.3 — 约束校验器（LLM 驱动 + 人工确认）

校验器不依赖关键词匹配或微调后的弱模型自评，而是调用**外部高质量 LLM（DeepSeek V4 Pro / GPT-5.5）** 对生成结果做结构化评估。此模块封装为独立 Python 模块 + 可选 **Claude Code Skill**（`/validate-chapter`）。

**核心设计原则**: 全自动校验在生成阶段易积累误差，采用**半自动人机协作**模式——LLM 发现问题 + 生成建议，人做最终决策。

| 校验项 | 方法 | 处理 |
|--------|------|------|
| 字数范围 | `len(chapter)` 统计中文字符（规则） | <2000 自动续写，>3000 截断 |
| 高潮嵌入 | LLM 判断：当前章节是否构成有效高潮/爽点，并标注高潮类型（打脸/逆袭/公布/获奖等） | 高潮不足 → 生成"高潮注入建议"→ 局部重写 |
| 人物一致性 | NER 抽取人物名 + 与知识库比对（规则+LLM 复核） | 不一致 → 自动修正或标记 |
| 金手指一致性 | LLM 判断：生成内容是否超出金手指设定范围 | 超出 → 标记违规段落 + 警告 |
| 重复检测 | n-gram 重叠度检测 + LLM 判断是否"换汤不换药"地在重复情节 | 重复率过高 → 要求重写变体 |
| 三线覆盖 | LLM 按"事业线/感情线/日常线"三维打分（0-10） | 某条线缺失（<3 分）→ 生成补充建议 |
| 结尾钩子 | LLM 判断最后 3 句是否制造了悬念/期待 | 钩子不足 → 生成 3 个备选钩子句 |
| 整体质量 | LLM 对章节做 1-5 分综合评分（文风+节奏+爽感+连贯性） | 低于 3 分 → 标记为需重写，输出具体问题 |

**CLI 交互模式——人工确认环节**（不强制但推荐在关键章节使用）:

```
[自动校验] 第 5 章: 高潮评分 4/10 (不足) | 三线覆盖: 事业8/日常7/感情1 ⚠️

[建议] 感情线缺失。建议在以下位置插入 2-3 句:
  - 场景A（片场收工后）: 女主角发来消息，主角看了一眼，嘴角微扬...
  - 场景B（票房庆功时）: 人群中，主角下意识搜寻那个身影...

[操作] (a) 接受建议自动插入  (b) 手动编辑  (c) 忽略继续  (d) 仅标记不修改
```

这种半自动模式比全自动更实用，降低了 LLM 误判导致的质量劣化风险，同时展示了人机协作的产品设计思考（面试加分项）。在非交互模式（批量生成）下，低于阈值的校验项自动标记为 `⚠️ 需人工复核` 而非强制修改。

**非交互模式的重试上限**（防止单章无限卡死）:
1. 生成后校验不通过 → LLM 输出修改建议 → 自动重写 → 再次校验
2. **单章最多重试 3 次**，第 3 次仍不通过则标记 `⚠️ 需人工复核`，保留最佳版本继续下一章
3. 全本生成完成后输出"问题章节汇总表"，供人工集中处理
4. 避免全自动死循环：重试过程中质量分不再提升（连续两次差值 <0.5）→ 提前终止重试

## 后续迭代方向

### GRPO 强化学习对齐网文偏好

由于 2 个月周期限制，GRPO 强化学习暂不纳入本次实施范围。但其奖励函数设计和训练架构已提前规划，即可以作为面试中展示 RL 理解深度的素材，也为项目后续迭代提供了明确的路线图。

**为什么选择 GRPO 而非 PPO**:
- GRPO 不需要 Value Network，节省显存
- 组内相对比较（group relative），方差更小，训练更稳定
- DeepSeek-R1 使用的正是 GRPO，技术路线经过验证

**奖励函数设计**（5 维加权，权重可根据消融实验调整）:

```python
def compute_reward(chapter, context):
    # 1. 文风匹配 (0.25) — 用 LLM（DeepSeek V4 Pro）打分替代分类器
    style_score = llm_score(chapter, rubric="口语化程度、梗密度、节奏感")
    
    # 2. 结构合规 (0.25)
    structure_score = (
        word_count_ok(chapter, 2000, 3000) * 0.4 +
        has_climax(chapter) * 0.3 +
        has_hook(chapter) * 0.3
    )
    
    # 3. 连贯性 (0.20)
    coherence_score = (
        entity_consistency(chapter, context.characters) * 0.5 +
        plot_continuity(chapter, context.prev_summary) * 0.5
    )
    
    # 4. 多样性 (0.20)
    diversity_score = 1.0 - repetition_ratio(chapter, context.all_chapters)
    
    # 5. 都市文娱专项 (0.10)
    urban_ent_score = llm_score(chapter, rubric="爽感、有梗、行业细节真实")
    
    return weighted_sum(style, structure, coherence, diversity, urban_ent)
```

**GRPO 训练配置**（当硬件条件允许时执行）:
```yaml
grpo:
  group_size: 4               # 每个 prompt 生成 4 个候选
  kl_penalty: 0.05            # 防止偏离 SFT 太远
  learning_rate: 1e-5         # RL 阶段学习率更低
  max_new_tokens: 3072
  temperature: 0.8
  episodes: 500~1000
  reward_normalization: z_score
```

**实施前置条件**:
- SFT 模型生成质量稳定（LLM 综合评分 ≥3.5/5）
- 奖励函数的 LLM 打分与人工偏好相关性验证通过（Pearson r ≥0.6）
- 硬件升级（建议 ≥2×H100 或 4×A6000）或租用云端 GPU

## 验证方案

### 各阶段验收标准

总体评估权重：**自动指标 : 人工评估 = 7 : 3**（自动化可复现 + 人工判断兜底）

| 阶段 | 验收标准 | 验证方式（自动70% + 人工30%） |
|------|---------|---------|
| Phase 1 | 清洗语料 ≥30 本都市文娱小说，SFT 样本 ≥6000 条，LLM 标注一致率: 客观任务 ≥85%，主观任务 ≥75% | 自动: 统计脚本 + LLM 标注一致率计算 / 人工: 抽检 300 条标注 |
| Phase 2 | Perplexity ↓ ≥15%，LLM 质量评分均值 ≥3.5/5 | 自动: wandb + DeepSeek V4 Pro 自动评分 / 人工: 盲评 20 组生成样本 |
| Phase 3 | RAG 检索 Top-5 场景匹配率 ≥70% | 自动: 标注测试集 precision@5 / 人工: 抽检 50 条检索结果 |
| Phase 4 | 字数命中率 ≥80%，LLM 高潮评估分均值 ≥6/10，人设一致性 ≥90%，LLM 综合质量分 ≥3.5/5 | 自动: 规则校验 + LLM 校验器自动打分 / 人工: 抽检 10 章完整输出 |
| Phase 5 | CLI 完整生成 50 章，Gradio 可交互，README 含复现步骤，读者可读性评分均值 ≥3.5/5 | 自动: 端到端运行测试 / 人工: 3 位网文读者做可读性评分 |

### 端到端测试流程

1. 输入 300 字梗概（含主角设定、金手指类型、时代背景）
2. 自动生成 50 章完整都市文娱小说
3. 自动校验（权重 70%）:
   - 黄金三章结构: ch1 重生+金手指 ✅, ch2 初试 ✅, ch3 打脸+钩子 ✅（LLM 判断）
   - 高潮节奏: ch5/10/15/20... 高潮覆盖率，LLM 逐章打分
   - 字数合规: 每章 2000-3000 字（容差 ±200），规则统计命中率
   - 人设一致性: 主角名/金手指/重要配角前后比对
   - 三线覆盖: LLM 逐章评估事业线/感情线/日常线得分
   - 综合质量: LLM 对整本书做 1-5 分质量评估
4. 人工评估（权重 30%）:
   - 3 位有网文阅读经验的读者做可读性评分（1-5 分，目标均值 ≥3.5）
   - 关键章节（ch1-3 黄金三章 + ch5 小高潮 + ch10 大高潮）人工通读评价
5. 导出 txt/epub，README 中可完整复现
