# 网文生成大模型 — 项目执行规划

## Context

**目标**: 构建基于大模型的都市文娱小说生成系统，作为 AI 岗位实习申请作品集。通过 SFT 微调使基座模型掌握都市文娱的文风与叙事模式，结合双路 RAG（文风参考 + 知识检索）增强生成质量，通过约束系统强制执行网文结构规范（黄金三章、单元化叙事、高潮节奏）。全程使用高质量外部 LLM（DeepSeek V4 Flash/Pro）辅助数据标注、内容分类和生成校验。

**核心决策**:

| 决策项 | 选择 |
|--------|------|
| 基座模型 | **Qwen3-30B-A3B**（30B 总参 / 3B 激活 MoE，中文能力强） |
| 题材 | **都市-文娱**（娱乐圈/导演/歌手/文抄公等重生或系统流） |
| 微调方式 | LoRA（rank=64, alpha=128），bf16 原生精度，单卡 RTX 6000D 80GB |
| RAG | 双路：文风参考 + 结构化知识（人物/作品/行业梗）|
| 数据标注 | DeepSeek V4 Flash 自动标注 + 人工复核 300 条 |
| 约束系统 | ✅ 大纲规划器 + 章节生成器 + LLM 校验器（已实现）|
| 硬件 | AutoDL RTX 6000D 80GB（¥5.18/h）|
| 周期 | 2 个月 |

- ✅ Phase 1: 数据管线（采集/清洗/标注/SFT样本/人工复核）

## 都市文娱题材特征分析

### 文风特点
- **轻松诙谐、爽感驱动**: 语言口语化、有梗、节奏快，拒绝沉重说教
- **半架空与现实交织**: 要么是华娱（真实明星/事件），要么是平行世界（文抄公流）
- **专业感+接地气**: 行业细节要有真实感（如拍电影流程、综艺录制），但人物行为要有人情味

### 叙事结构
- **单元化叙事**: 以"项目"为单位（拍一部电影、上一档综艺、发一张专辑），每个单元有完整的铺垫→高潮→收尾
- **三线并进**: 事业线（文娱成就）+ 感情线（恋爱/修罗场）+ 日常线（轻松互动/配角情景剧）
- **金手指模式**: 重生先知、系统任务、文抄公是主流，金手指需"模糊化"处理（不详细解释机制）

### 黄金三章（都市文娱版）
1. **第一章**: 穿越/重生设定 + 当前困境（如：平行世界的落魄歌手/导演）+ 金手指激活
2. **第二章**: 金手指初试（如：搬运第一首歌/第一部电影）+ 小范围引起关注
3. **第三章**: 第一次冲突/打脸 + 更大机遇降临 + 悬念钩子

### 高潮节奏（都市文娱版）
- **每 5 章小高潮**: 一个"单元"完结（一部作品发布/一次综艺播出/一次打脸成功）
- **每 10 章大高潮**: 重大事业突破（获奖/票房大爆/行业地位跃升）+ 感情线重大进展

- **清洗**: 编码自动检测（UTF-8/GBK/GB18030）、章节正则切分、广告水印过滤
- **6 个章节识别 bug 修复**:
  1. `^\d+[、，.\s]+.*` 匹配了"1.5亿票房"等数字 → 移除 `[，.\s]`
  2. 字符集 `[章回节卷集]` 包含"集"→ "第一集收视率"误识别 → 移除"集"
  3. 字符集包含"节"→ "第一节课还没开始"误识别 → 移除"节"
  4. 句末标点未检查 → "第一章的内容并不复杂。"误识别 → 添加句末标点检查
  5. 章节标记后紧跟标点 → "第2章，写不出来了"误识别 → 正则排除
  6. 中文数字解析器 broken → "十一"返回1而非11 → 重写 `_parse_chinese_number`
- **产出**: `data/clean/` 下 2,104 章清洗后章节 + metadata.json

### 任务 1.3 — LLM 标注

- **模型**: DeepSeek V4 Flash（`deepseek-v4-flash`）
- **5 个标注维度**: 场景类型（6类）、叙事节奏（5类）、实体抽取、三线覆盖评分（0-10）、质量评分（1-5）
- **全量标注**: 2,104 章，费用 ~$1.50
- **人工复核**: 300 条抽检，100% 一致率
- **产出**: `data/clean/*/llm_annotations.json`

### 任务 1.4 — SFT 训练数据构造

- **格式**: 使用 Qwen3 原生 tokenizer 的 `apply_chat_template()` 自动生成，不手写 ChatML
- **5 种任务类型**:

| 任务类型 | 占比 | 构造方法 |
|---------|------|---------|
| 章节续写 | 30% | 同书连续章节，前 N-1 章为 prompt，第 N 章为 target |
| 单元创作 | 25% | 利用节奏标注识别单元边界（5章一单元），单元大纲为 prompt |
| 场景生成 | 20% | LLM 标注的场景标签 + 前 200 字情境 → 场景全文 |
| 风格模仿 | 15% | 同作者 3 段短文 few-shot → 另一段为 target |
| 大纲扩展 | 10% | 利用节奏标注构造简易大纲 → 章节全文 |

- **产出**: `data/sft/train.jsonl` (5,400 条) + `val.jsonl` (600 条)

## Phase 5: GRPO 强化学习（第 7-8 周）

### 为什么加入 GRPO

SFT 让模型学会了网文风格（对话占比 0.1%→32.7%），但有一个根本限制：**模型只学到了"模仿"，没有学到"什么是好的"**。GRPO 让模型在生成→评估→优化的循环中自我改进：

```
SFT 模型对同个 prompt 生成 2 个候选
    ↓
DeepSeek API 给每个候选打分（文风/爽感/结构/连贯性/多样性）
    ↓
GRPO 对比组内相对得分 → 好的被强化，差的被抑制
    ↓
循环 500 轮 → 模型越来越懂"什么是好的网文"
```

面试叙事：**"SFT 让模型学会写网文，GRPO 让模型学会写好网文"**——RLHF 在垂直领域的完整落地，这是绝大多数实习生项目做不到的。

### 硬件可行性：RTX 6000D 80GB 跑 GRPO

GRPO 相比 PPO 不需要 Value Network，同硬件就能跑：

```
显存预算 (80GB):
  基座模型 (bf16, 冻结):        60GB  ← 只推理不更新
  LoRA 适配器 (可训练):        ~0.5GB  ← rank=32
  参考模型 (梯度关闭, 共享基座):  0GB  ← 复用基座
  生成 2 个候选 (2048 tokens):  ~2GB  ← group_size=2
  奖励计算:                      0GB  ← DeepSeek API, 不占显存
  优化器 + 激活值:             ~10GB
  ────────────────────────────────
  合计:                        ~73GB  ✅ 留 7GB 余量
```

**备选方案**: RTX PRO 6000 96GB（¥5.98/h）可开 group_size=4，收敛更快，多花 ¥5/次训练。

### 奖励函数（5 维加权 + DeepSeek API 打分）

```python
def compute_reward(chapter, context):
    # 1. 文风匹配 (0.25) — DeepSeek API 打分
    style = llm_score(chapter, "口语化程度、梗密度、节奏感")

    # 2. 结构合规 (0.25) — 规则 + API 混合
    wc = len(chapter)
    structure = (
        (1 if 2000<=wc<=3000 else 0.5 if abs(wc-2500)<1000 else 0) * 0.4 +
        (0.8 if has_climax(chapter) else 0.2) * 0.3 +
        (0.8 if has_hook(chapter) else 0.2) * 0.3
    )

    # 3. 连贯性 (0.20) — 人物一致性 + 情节连续性
    coherence = (
        entity_consistency(chapter, context.characters) * 0.5 +
        plot_continuity(chapter, context.summary) * 0.5
    )

    # 4. 多样性 (0.20) — 4-gram 重叠率越低越好
    diversity = 1.0 - max(ngram_overlap(chapter, ch) for ch in context.previous)

    # 5. 都市文娱专项 (0.10) — DeepSeek API 打分
    entertainment = llm_score(chapter, "打脸爽感、行业细节、有网络梗")

    return 0.25*style + 0.25*structure + 0.20*coherence + 0.20*diversity + 0.10*entertainment
```

### 训练配置

```yaml
model:
  base: Qwen3-30B-A3B (bf16, 冻结)
  lora: rank=32, alpha=64 (SFT 权重初始化)

grpo:
  group_size: 2             # 每个 prompt 生成 2 个候选
  kl_penalty: 0.05          # 防止偏离 SFT 太远
  learning_rate: 1e-5       # SFT 的 1/20
  max_new_tokens: 2048
  temperature: 0.8          # 适度探索
  total_episodes: 500        # 500 个 prompt × 2 候选
  reward_normalization: z_score

hardware:
  gpu: 1× RTX 6000D 80GB
  cost: ¥5.18/hour
  estimated_time: 6-10 hours
  estimated_cost: ¥35-50
```

| 指标 | 目标 |
|------|------|

### 面试价值

| 面试官会问 | 回答 |
|-----------|------|
| 为什么 GRPO 不用 PPO | 不需要 Value Network，80GB 单卡跑。组内相对比较比绝对打分稳定 |
| 奖励函数怎么设计 | 5 维加权 + DeepSeek API 打分，权重可消融验证 |
| 怎么防止 reward hacking | KL 惩罚 + LLM 打分（难 exploit）+ z-score 归一化 |
| 稳定吗 | SFT 权重初始化 + 梯度裁剪 + early stopping |
